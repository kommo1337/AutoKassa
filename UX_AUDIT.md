# UX-аудит AutoKassa — Полный отчёт

## Контекст
AutoKassa — WPF-приложение для учёта финансов автосервиса. 14 экранов, MVVM + EF Core + SQLite.
Целевой пользователь: администратор/бухгалтер автосервиса, работает за десктопом, вносит операции в реальном времени.

Аудит проведён по всем View/ViewModel/Service файлам проекта. Ниже — карта сценариев, найденные проблемы и предложения.

## 🔴 КРИТИЧНО — пользователь страдает прямо сейчас

### 1. Потеря данных при закрытии формы операции
**Проблема:** Пользователь заполнил сумму, выбрал категорию, написал описание → случайно нажал Esc или кнопку отмены → **все данные потеряны без предупреждения**. `CancelCommand` просто закрывает модалку.
**Влияние:** Фрустрация, повторный ввод. Особенно болезненно для длинных описаний.
**Решение:** Добавить проверку `HasUnsavedChanges` в `TransactionEditViewModel`. Если форма изменена — показать подтверждение «Отменить изменения?» перед закрытием. Аналогично для inline-добавления в TransactionsView.
**Файлы:** `TransactionEditViewModel.cs`, `TransactionsViewModel.cs`

### 2. Нет обратной связи после сохранения операции
**Проблема:** Пользователь нажимает «Добавить расход» в модалке → модалка закрывается → **никакого подтверждения**. QuickAdd показывает toast «Операция добавлена успешно», а полная форма — нет.
**Влияние:** Пользователь не уверен, сохранилась ли операция. Может добавить дубликат.
**Решение:** Добавить `_toastService.ShowSuccess("Операция сохранена")` после успешного `SaveAsync()` в `TransactionEditViewModel.SaveAsync()`.
**Файлы:** `TransactionEditViewModel.cs`

### 3. Валидация через блокирующие MessageBox
**Проблема:** Ошибки валидации в Настройках (время блокировки, размер страницы и т.д.) показываются через `_dialogService.ShowWarning()` — это модальный `MessageBox`, который блокирует всё приложение. Пользователь не видит, какое поле неправильное.
**Влияние:** Пользователь должен закрыть MessageBox, вспомнить ошибку, найти поле, исправить.
**Решение:** Заменить MessageBox на inline-валидацию: красная рамка поля + текст ошибки под полем. Использовать `INotifyDataErrorInfo` (уже есть в ViewModelBase) + `Validation.ErrorTemplate` в XAML.
**Файлы:** `SettingsViewModel.cs`, `SettingsView.xaml`


### 5. Пустые состояния отсутствуют на ключевых экранах
**Проблема:** 
- **Операции** — при пустом результате фильтрации показывается **пустое пространство** без объяснения
- **Категории** — пустой список без подсказки
- **Отчёты** — нет сообщения когда за период нет данных
**Влияние:** Пользователь не понимает — данных нет, или что-то сломалось?
**Решение:** Добавить шаблоны пустых состояний с иконкой + текстом + действием:
- Операции: «Операций не найдено. Попробуйте изменить фильтры или добавьте первую операцию»
- Категории: «Нет категорий. Создайте первую категорию»
- Отчёты: «Нет данных за выбранный период»
**Файлы:** `TransactionsView.xaml`, `CategoriesView.xaml`, `BalanceReportView.xaml`, `CategoryReportView.xaml`, `TransactionDetailReportView.xaml`

## 🟡 ВАЖНО — заметно улучшит опыт

### 6. Нет онбординга для новых пользователей
**Чего не хватает:** После установки пароля пользователь попадает на пустой дашборд без единой подсказки. Не знает что делать: куда нажимать, как добавить категорию, что означают карточки.
**Почему это нужно:** Автосервис — не IT-компания. Пользователь может быть далёк от технологий.
**Решение:** Добавить «Getting Started» чеклист на Dashboard при пустой БД:
- ✅ Установить пароль (уже сделано)
- ⬜ Настроить категории
- ⬜ Добавить первую операцию
- ⬜ Посмотреть отчёты
Каждый пункт — ссылка на соответствующий экран. Скрывается после выполнения всех шагов (или по кнопке «Скрыть»).

### 7. Поиск операций не в реальном времени
**Чего не хватает:** Поиск по описанию требует нажатия Enter или кнопки «Применить фильтры». Пользователь печатает и ждёт, ничего не происходит.
**Почему это нужно:** Все современные приложения имеют instant search.
**Решение:** Добавить debounce-поиск (300-500ms задержка после последнего ввода) в `TransactionsViewModel`. Использовать `CancellationTokenSource` для отмены предыдущего запроса.
**Файлы:** `TransactionsViewModel.cs`

### 8. Нет экспорта с экрана Операций
**Чего не хватает:** Пользователь отфильтровал операции (например, «все расходы наличными за март») → хочет выгрузить → **нет кнопки экспорта**. Нужно идти в Отчёты, заново настраивать фильтры, формировать отчёт, и только потом экспортировать.
**Почему это нужно:** Экспорт — это действие над текущей выборкой, а не отдельный процесс.
**Решение:** Добавить кнопку «Экспорт» (PDF/Excel) прямо на панель фильтров в TransactionsView. Экспортировать текущий отфильтрованный набор.
**Файлы:** `TransactionsView.xaml`, `TransactionsViewModel.cs`, `ExportService.cs`

### 9. Нет массового редактирования
**Чего не хватает:** Можно массово удалить операции (чекбоксы + «Удалить выбранные»), но нельзя массово:
- Сменить категорию (например, перенести 20 операций из «Прочее» в «Запчасти»)
- Сменить тип оплаты
**Почему это нужно:** При ошибке категоризации приходится редактировать каждую запись по одной.
**Решение:** Панель массовых действий: при выборе ≥2 операций показать «Изменить категорию» / «Изменить тип оплаты» / «Удалить».
**Файлы:** `TransactionsView.xaml`, `TransactionsViewModel.cs`

### 10. Нет Undo для редактирования
**Чего не хватает:** Toast с Undo есть только для удаления. Если пользователь случайно изменил сумму или категорию и сохранил — нет способа откатить. Нужно вспомнить, что было, и вручную исправить.
**Почему это нужно:** Ошибки при редактировании — частая ситуация.
**Решение:** При сохранении изменений в `TransactionEditViewModel` — сохранять снапшот предыдущего состояния и показывать toast «Операция изменена — Отменить». По undo — восстанавливать снапшот через `UpdateAsync()`.
**Файлы:** `TransactionEditViewModel.cs`, `TransactionsViewModel.cs`, `DashboardViewModel.cs`

### 11. Нет индикатора загрузки в UI
**Чего не хватает:** Все ViewModels имеют свойство `IsLoading`, но ни одна View не привязывает к нему спиннер или оверлей. При медленной БД пользователь видит пустой экран.
**Почему это нужно:** Без спиннера пользователь думает, что приложение зависло.
**Решение:** Добавить общий шаблон загрузки (ProgressRing / spinner) с привязкой к `IsLoading` на каждом экране.
**Файлы:** `DashboardView.xaml`, `TransactionsView.xaml`, `CategoriesView.xaml`, все ReportView файлы

### 13. Нет подсказки в форме валидации (нет красных рамок)
**Чего не хватает:** Единственная визуальная валидация — красный текст ошибки под полем суммы в TransactionEditView. Нет красной рамки на невалидных полях, нет выделения обязательных полей.
**Почему это нужно:** Пользователь не понимает, почему кнопка «Сохранить» неактивна.
**Решение:** Добавить `Validation.ErrorTemplate` с красной рамкой + `*` у обязательных полей.
**Файлы:** `TransactionEditView.xaml`, `App.xaml` (общий стиль)

## 🟢 БЫЛО БЫ КРУТО — конкурентное преимущество

### 16. Горячие клавиши для частых действий
**Идея:** Сейчас есть только Alt+1-4 и Ctrl+L. Добавить:
- `Ctrl+N` → Новая операция (глобально, с любого экрана)
- `Ctrl+F` → Фокус на поиск в Операциях
- `Ctrl+E` → Экспорт
- `Delete` → Удалить выбранное
**Задачу решает:** Ускоряет работу опытных пользователей.
**Объём:** InputBindings в MainWindow + роутинг команд.

### 17. Статистика на дашборде: сравнение с предыдущим периодом
**Идея:** Рядом с суммой доходов/расходов — стрелка ↑↓ и процент изменения vs прошлый аналогичный период (прошлая неделя, прошлый месяц).
**Задачу решает:** Владелец сервиса видит динамику: бизнес растёт или падает.
**Объём:** Дополнительный запрос в `DashboardViewModel` + UI-элемент в карточках.
